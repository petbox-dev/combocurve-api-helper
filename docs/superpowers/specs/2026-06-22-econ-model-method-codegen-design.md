# Econ-model method codegen for combocurve-api-helper

Date: 2026-06-22
Status: design (awaiting review)

## Problem

`models.py` hand-writes ~100 per-type methods — for each econ-model type:
`get_<seg>_models[_url]`, `get_<seg>_model_by_id[_url]`,
`get_<seg>_assignments_by_id[_url]` — each a thin delegation to a generic
(`get_econ_models_by_type*`, `get_econ_model_by_type_by_id*`,
`get_econ_model_assignments_by_type_by_id*`) with a hard-coded `econModelType`.

Consequences:
- **Drift.** The hand-maintained projection of `econModels.json` drifts from it:
  `operations` was missing from the JSON; two names are irregular
  (`get_fluid_models_by_id_url` vs the `_model_by_id_url` norm;
  `get_reserves_categories_by_id` vs the `_model_by_id` norm); `GeneralOptions`
  has CRUD methods but no assignments; null-route types have neither.
- **No assignment writes.** The REST route
  `POST/PUT/DELETE /projects/{id}/econ-models/{econName}/{modelId}/assignments`
  exists, but the helper exposes only `GET`. Callers (`cc-afe-sync`,
  `cc-insert-wells`) reach it via a raw session + the `_url` builder.
- **Boilerplate.** ~1000 lines of near-identical methods; every new type or
  operation means editing N methods.

(Separately, the scenario assignment-grid GET — `/scenarios/{id}/assignments/
econ-models` — had no wrapper; `get_scenario_econ_model_assignments` was added
by hand to `scenarios.py`. It is a scenario method, not per-type, so it is out
of codegen scope; it is recorded here under Rollout.)

## Goal

Make `econModels.json` the single source of truth that drives **generated,
typed, consistent** per-type methods (reads **and** assignment writes),
eliminating drift and boilerplate and closing the write gap — with no runtime
metaprogramming (generated code is real, static, type-checkable source).

## Design

### 1. Source of truth — `assets/econModels.json`

Each entry gains explicit capability flags and a method-name base:
```json
{ "qualifier": "capex", "econModelType": "Capex", "route": "capex",
  "methodBase": "capex", "hasCrud": true, "assignable": true }
```
- `hasCrud` = `route != null` (the type has a model CRUD route).
- `assignable` = explicit boolean. Set to the types that currently expose
  assignment methods (15) **plus `Operations`** (which has no methods today but
  is assignable — confirmed live). `GeneralOptions` and the null-route types
  (`Forecast`, `Network`, `PSeries`, `Schedule`) are **not** assignable.
- `methodBase` = the method-name base segment, stored **explicitly** rather than
  derived. It equals `route.replace('-', '_')` for every type **except
  `FluidModel`, where it is `fluid`** (route-snake `fluid_models` would yield
  ugly `get_fluid_models_models`). Storing it avoids a derivation special-case
  in the generator. Null-route types have no `methodBase` → no per-type methods
  (correct: `forecast` is not a normal econ-model resource).

The three name spaces remain distinct and are all carried by the JSON: the
method-name segment is route-derived (snake); the assignment URL uses
`econModelType` (PascalCase); the CRUD URL uses `route` (kebab). The per-type
methods pass `econModelType` to the generics, which map to the right namespace.

### 2. Generator — `scripts/generate_model_methods.py`

Reads `econModels.json`; for each entry emits, gated by flags:
- `hasCrud` → `get_<seg>_models[_url]`, `get_<seg>_model_by_id[_url]`
- `assignable` → `get_<seg>_assignments_by_id[_url]` **and**
  `post_/put_/delete_<seg>_assignments_by_id`

Output: `src/combocurve_api_helper/_models_generated.py` — real, typed source
with a `# DO NOT EDIT — generated by scripts/generate_model_methods.py` header,
defining a mixin class `_GeneratedModelMethods`. Each method is a typed
signature + one-line docstring that delegates to a generic. Run manually;
output is committed.

### 3. Generic primitives — `models.py` (or a shared base)

- Existing GET generics stay.
- **New write generics:** `post_/put_/delete_econ_model_assignments_by_type_by_id
  (project_id, econ_model_type, model_id, data)`, built on `_post_items` /
  `_put_items` / `_delete_responses` and the existing
  `get_econ_model_assignments_by_type_by_id_url` builder.
- Position the generics so the generated mixin type-checks when calling them —
  either a shared base both `Models` and `_GeneratedModelMethods` build on, or a
  `Protocol` the mixin is typed against. (Mechanism finalized in the plan.)

### 4. Composition

`class Models(_GeneratedModelMethods, APIBase)` — generics + write primitives in
`Models`, per-type methods inherited from the generated mixin.
`ComboCurveAPI(Root, …, Models, …)` MRO is unchanged. The dev test constructs
the client via `ComboCurveAPI.from_alternate_config(combocurve_json_path,
cc_api_config_json_path)` pointed at `~/.combocurve/dev/`.

### 5. Back-compat — normalize + fix callers

Six method names change to the consistent scheme (four singular `_models`
fetches that should be plural, plus two other irregulars):

| Current (irregular) | Normalized |
|---|---|
| `get_differential_models` | `get_differentials_models` |
| `get_emission_models` | `get_emissions_models` |
| `get_expense_models` | `get_expenses_models` |
| `get_risking_models` | `get_riskings_models` |
| `get_fluid_models_by_id_url` | `get_fluid_model_by_id_url` |
| `get_reserves_categories_by_id` | `get_reserves_categories_model_by_id` |

`FluidModel`'s base stays `fluid` (per `methodBase`), so the other fluid methods
keep their names. `Operations` gains a full method set (additive, no breakage).
No aliases. Grep `cc-afe-sync`, `cc-insert-wells`, `vdr` (and the helper's own
code) for the six old names and fix usages.

### 6. Verification

**Offline (runs anywhere, no creds):**
- Method-set test: the public method names on `_GeneratedModelMethods` equal the
  set derived from `econModels.json` + the flags (catches drift in either
  direction).
- Freshness test: re-run the generator into a temp buffer and assert it is
  byte-identical to the committed `_models_generated.py` (fails if someone edits
  the JSON without regenerating).
- Existing helper tests pass; `ruff` + `mypy` clean.

**Live (dev, skipped by default — gated on `~/.combocurve/dev` creds present):**
- Construct via `from_alternate_config` against dev.
- In an existing dev project + scenario (with wells), create a uniquely-named
  throwaway econ model (e.g. capex `zz_apitest_<ts>`), `post` an assignment to a
  qualifier for a well, read it back via `get_<seg>_assignments_by_id`, assert
  the round-trip, then `delete` the model/assignment. Self-cleaning; minimal
  footprint. This is the real proof the write payload/route is correct.

### 7. Rollout

1. Bump `__version__`, rebuild, `twine` to internal PyPI; reinstall in the
   `core` env (the install is non-editable, so source edits don't reach
   consumers until reinstall).
2. Switch `vdr qc cc` from `api._get_items(url)` to
   `api.get_scenario_econ_model_assignments(project_id, scenario_id)`; drop the
   manual URL build.
3. Fix the two renamed-method callers across repos.

## Out of scope (possible follow-ups)

- `company_models.py` has a parallel per-type sprawl (`get_company_<type>_*`);
  the same generator could cover it later. Not in this pass.
- Per-type CRUD **writes** (`post_/put_/delete_<seg>_model`) — only assignment
  writes are added now; model CRUD stays via the existing generics unless a need
  appears.
